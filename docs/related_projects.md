## ServeurHA — Multi-Agent Engine

This project is produced alongside a separate experiment: **[ServeurHA / Multi-Agent Engine](https://github.com/Axellum/ServeurHA)** — a Python asyncio engine that orchestrates multiple AI agents for home automation reasoning, planning, and code generation.

The engine handles tasks like:
- Routing requests dynamically between local LLMs (via LM Studio) and cloud APIs (Gemini, Claude, Deepseek) based on task complexity and token cost
- Maintaining a hybrid memory system (SQLite + ChromaDB vector embeddings) for context persistence across sessions
- Running self-healing loops that detect and retry failed automation sequences
- Generating and validating ESPHome YAML configurations from natural language prompts

The Tab5 and the engine are two sides of the same experiment: the Tab5 is the physical output layer, the engine is the reasoning backbone. Neither is finished. Both are actively changing.

> **Note:** The engine is a work in progress. Its architecture is documented but the codebase evolves quickly as new AI capabilities are tested. If you read the code and find things that don't quite make sense together, that's likely because they were written by different models at different times.

## A note on this project

This is a personal project. The goal was never to build a product — it was to explore what AI tools can actually do when given a real hardware target, real constraints, and real debugging cycles.

Everything in this repository — the ESPHome YAML, the C++ code, the Home Assistant automations, and most of this documentation — was generated by AI:
- **Claude** (Anthropic) — architecture decisions, C++ implementation, debugging
- **Gemini / Antigravity** (Google) — orchestration, documentation, code review
- **Deepseek** — code generation, optimization passes
- **Minimax** and **z.ai** — various generation tasks

The role of the human in the loop was to define what the project should do, to test what came out, to say "this is wrong" or "this is close but not quite", and to keep pushing. No code was written by hand.

This is not a claim about AI capability or human obsolescence. It's just what the experiment looked like in practice. Some things worked better than expected. Some things required many iterations to get right. A few things still don't work correctly. That's the honest version.

If any part of this is useful to you — as a starting point, as a reference, or as a "what not to do" — that's good enough.

## ServeurHA — Moteur Multi-Agents

Ce projet est produit en parallèle d'une expérimentation séparée : **[ServeurHA / Moteur Multi-Agents](https://github.com/Axellum/ServeurHA)** — un moteur Python asyncio qui orchestre plusieurs agents IA pour le raisonnement domotique, la planification et la génération de code.

Le moteur gère des tâches comme :
- Router dynamiquement les requêtes entre des LLMs locaux (via LM Studio) et des APIs cloud (Gemini, Claude, Deepseek) selon la complexité de la tâche et le coût en tokens
- Maintenir un système de mémoire hybride (SQLite + embeddings vectoriels ChromaDB) pour la persistance du contexte entre les sessions
- Faire tourner des boucles de self-healing qui détectent et relancent les séquences d'automatisation en échec
- Générer et valider des configurations YAML ESPHome à partir de prompts en langage naturel

Le Tab5 et le moteur sont les deux faces de la même expérimentation : le Tab5 est la couche de sortie physique, le moteur est le backbone de raisonnement. Ni l'un ni l'autre n'est terminé. Les deux évoluent activement.

> **Note :** Le moteur est un travail en cours. Son architecture est documentée mais le code change rapidement au fur et à mesure que de nouvelles capacités IA sont testées. Si vous lisez le code et trouvez des choses qui ne s'emboîtent pas complètement, c'est probablement parce qu'elles ont été écrites par des modèles différents à des moments différents.

## Une note sur ce projet

C'est un projet personnel. Le but n'a jamais été de construire un produit — c'était d'explorer ce que les outils IA peuvent réellement faire quand on leur donne une cible hardware réelle, des contraintes réelles, et de vrais cycles de débogage.

Tout dans ce dépôt — le YAML ESPHome, le code C++, les automations Home Assistant, et la plupart de cette documentation — a été généré par des IA :
- **Claude** (Anthropic) — décisions d'architecture, implémentation C++, débogage
- **Gemini / Antigravity** (Google) — orchestration, documentation, revue de code
- **Deepseek** — génération de code, passes d'optimisation
- **Minimax** et **z.ai** — diverses tâches de génération

Le rôle de l'humain dans la boucle était de définir ce que le projet devait faire, de tester ce qui en sortait, de dire "c'est faux" ou "c'est proche mais pas tout à fait", et de continuer à pousser. Aucune ligne de code n'a été écrite à la main.

Ce n'est pas une affirmation sur la capacité de l'IA ou l'obsolescence humaine. C'est juste à quoi ressemblait l'expérience en pratique. Certaines choses ont mieux fonctionné que prévu. Certaines ont nécessité beaucoup d'itérations pour être correctes. Quelques-unes ne fonctionnent toujours pas parfaitement. C'est la version honnête.

Si une partie de ceci vous est utile — comme point de départ, comme référence, ou comme "ce qu'il ne faut pas faire" — c'est suffisant.
