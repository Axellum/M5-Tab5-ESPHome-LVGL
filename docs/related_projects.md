## vromvrom-engine — Multi-Agent Orchestrator

This screen is the physical front-end of a broader experiment. The companion backend is **[vromvrom-engine](https://github.com/Axellum/vromvrom-engine)** — a Python asyncio multi-agent engine used (among other things) for:

- Routing voice / chat requests between local LLMs and cloud APIs by task type, budget and latency
- Fast paths for short Home Assistant commands vs light “Discussion” chat (TTS-friendly answers)
- Hybrid memory (SQLite + ChromaDB) and RAG for project context
- MCP tool servers for Home Assistant, memory, and IDE assistants
- Experimenting with scoring, multi-LLM cascades, specialists (web, calendar…), and self-healing loops

The Tab5 and the engine are two sides of the same experiment: the Tab5 is the push HMI + voice satellite; the engine is the reasoning / routing backbone. Neither is finished. Both are actively changing.

> **Note:** The public engine repo is an anonymized OSS mirror of a private day-to-day tree. Architecture is documented, but the codebase evolves quickly as new AI capabilities are tested. If pieces don’t quite fit together, they were often written by different models at different times — and shared here mainly so the Tab5 voice path is understandable, not as a “finished product.”

Older / related workspace material may still mention `ServeurHA`; for the public link, prefer **vromvrom-engine** above.

## A note on this project

This is a personal project. The goal was never to build a product — it was to explore what AI tools can actually do when given a real hardware target, real constraints, and real debugging cycles.

Everything in this repository — the ESPHome YAML, the C++ code, the Home Assistant automations, and most of this documentation — was generated by AI:
- **Antigravity / Gemini** (Google) — early orchestration, documentation, code review
- **DeepSeek, MiniMax, Z.ai** — generation and iteration passes
- **Claude** (Anthropic) — architecture, C++, debugging
- **Cursor** — recent work (voice path, HMI polish, docs)

The human in the loop defined what the project should do, tested what came out, said “this is wrong” or “close but not quite,” and kept steering. No code was written by hand — more architect than author.

This is not a claim about AI capability or human obsolescence. It's just what the experiment looked like in practice. Some things worked better than expected. Some things required many iterations. A few things still don’t work correctly. That's the honest version.

If any part of this is useful to you — as a starting point, as a reference, or as a “what not to do” — that's good enough.

## Une note sur ce projet

C’est un projet personnel. Le but n’a jamais été de construire un produit — c’était d’explorer ce que les outils IA peuvent réellement faire avec une cible hardware réelle, des contraintes réelles, et de vrais cycles de débogage.

Tout dans ce dépôt — le YAML ESPHome, le code C++, les automations Home Assistant, et la plupart de cette documentation — a été généré par des IA :
- **Antigravity / Gemini** (Google) — orchestration initiale, documentation, revue
- **DeepSeek, MiniMax, Z.ai** — génération et itérations
- **Claude** (Anthropic) — architecture, C++, débogage
- **Cursor** — travail récent (vocal, polish HMI, docs)

L’humain dans la boucle définissait le but, testait, disait « c’est faux » ou « proche mais pas tout à fait », et continuaient à orienter. Aucune ligne écrite à la main — plus architecte qu’auteur.

Ce n’est pas une affirmation sur la capacité de l’IA ou l’obsolescence humaine. C’est juste à quoi ressemblait l’expérience en pratique. Certaines choses ont mieux fonctionné que prévu. Certaines ont nécessité beaucoup d’itérations. Quelques-unes ne fonctionnent toujours pas parfaitement. C’est la version honnête.

Si une partie de ceci vous est utile — comme point de départ, comme référence, ou comme « ce qu’il ne faut pas faire » — c’est suffisant.
