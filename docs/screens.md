## Navigation

| 🏠 | Home — summary view |

## Home screen

The default screen shown at startup. Gives a quick overview of the house state.

**What's displayed:**
- Current time and date
- Indoor temperature and humidity (living room sensor)
- A summary weather line — a short sentence generated by a Home Assistant template sensor: *"Rain in 10 min"*, *"No rain expected"*, *"Possible showers"*
- Active weather alert banner (if present) — see Weather section below
- Microphone icon with pipeline state color (see Voice Assistant section)
- Mode toggle button (Home Assistant mode vs. conversation/LLM mode)

## Weather screen

The weather section is a single screen navigated by **horizontal swipe**, with a **5-slide pagination system** (indicated by 5 dots at the bottom of the screen). All slides share the same 5-card layout — 5 cards displayed side by side in the lower portion of the screen.

### Slides 1–2: Hourly forecast (15 time slots)

The first two slides cover **hourly weather** for the next 15 time slots, 5 slots per slide (slide 1 = slots 1–5, slide 2 = slots 6–10... up to slot 15).

Each card shows:
- **Time label** (e.g., `14:00`, `15:00`)
- **Weather condition icon** — layered dual-icon system: main condition (cloud, sun, moon, fog, wind) + precipitation overlay (rain, heavy rain, snow, hail, thunder) composited from the `IconeMeteo.ttf` font
- **Temperature** — color-coded: blue (cold) → green → orange → red (hot)
- **Rainfall** — precipitation in mm if > 0, `-` if dry

### Slides 3–5: 15-day daily forecast

The last three slides cover the **15-day daily forecast**, 5 days per slide.

Each card shows:
- **Day name** — color-coded by type:
  - Cyan = today
  - Green = day off / rest day
  - Amber/Red = Sunday (worked or off)
  - Rose = early shift (start before 09:00)
  - Dim slate = past day (greyed out opacity)
- **Weather condition icon** (same dual-layer system as hourly)
- **Max / Min temperatures** — both color-coded individually

Tapping a day card **toggles** between weather view and an opening-hours detail view (configured per day in the HA push payload).

**On slide 3 only (today + next 4 days):** each card also shows a shortcut action button (PC/TV, roller shutter, lights) — these disappear on slides 4 and 5 to keep the cards clean.

**Data source:** Météo-France integration in Home Assistant. HA serializes the full 15-day array into a single semicolon-delimited push payload per update.

## Climate screen

Controls the room's air conditioning unit via Home Assistant's `climate` entity.

**What's displayed:**
- Current room temperature (from a dedicated temperature sensor)
- Target temperature — shown as a large number in the center of an arc control
- AC operating mode (heating / cooling / fan only / auto / off) — displayed as a mode label with a matching icon

**What you can do:**
- **Arc thermostat** — drag the arc indicator to set the target temperature. The arc animates smoothly and sends the new target to Home Assistant in real time.
- **Mode buttons** — tap to switch between heating, cooling, auto, and off modes. The active mode button is highlighted.

The climate screen stays grayed out (all controls dimmed, not hidden) when the AC is turned off, preserving the layout while clearly communicating that the unit is inactive.

---

## Plants screen

Monitors up to five BLE soil moisture sensors (compatible with standard Xiaomi/MiFlora-style sensors).

**What's displayed per sensor slot:**
- Plant name label (customizable in the substitutions block)
- Soil moisture percentage — shown as a horizontal progress bar
- Bar color varies by moisture level:
  - 🔴 Red — very dry (< 20%) — needs watering
  - 🟠 Orange — dry (20–35%)
  - 🟢 Green — optimal (35–65%)
  - 🔵 Blue — too wet (> 65%)
- Current temperature from the plant area sensor

The BLE sensors are polled by the ESP32-C6 co-processor. Data flows through the ESPHome BLE proxy to Home Assistant, then gets pushed back to the screen via the standard push automation. No direct BLE-to-screen path.

## Planning screen

Displays the next upcoming calendar events from Google Calendar.

**What's displayed:**
- Up to 4 event slots
- Each slot shows:
  - Event title (truncated if too long)
  - Start time or date
  - A color tag matching the Google Calendar event color

The color tags use LVGL's inline color syntax — Home Assistant's template reads the event color from the Google Calendar API and embeds it as a hex code directly in the push payload string. The device renders it without any additional color logic.

Data updates on a `calendar.sync` trigger, or when the calendar entity state changes. The planning screen does not refresh on a timer — it waits for a push.

## Voice assistant

The microphone icon on the home screen is the visual interface for the voice assistant. It changes color to reflect the current pipeline state:

| Icon color | State | What's happening |
|-----------|-------|-----------------|
| **Dim grey** | Standby | Wake-word listening is disabled |
| **Grey** | Idle | Listening for "Ok Nabu" in the background |
| **Green** | Listening | Wake word detected — recording your speech |
| **Orange** | Processing | HA pipeline is running STT + intent recognition |
| **Blue** | Speaking | TTS response is playing through the speaker |
| **Red** | Error | Pipeline failed or command not understood |

**Wake-word toggle:** a button on the home screen activates or deactivates the "Ok Nabu" wake-word detection. When off, you can still activate the voice assistant by tapping the microphone icon directly (push-to-talk). The icon turns dim grey when detection is off.

**Mode selector:** next to the microphone icon, a small button toggles between two modes:
- 🏠 **Home Assistant mode** — commands go to the standard HA conversation agent (control lights, check sensors, trigger automations)
- 🤖 **Conversation mode** — commands go to an LLM-backed pipeline for free-form conversation

The mode is saved across reboots via the HA `select` entity (`select.m5stack_tab5_home_assistant_hmi_assistant`).

## Console screen

A scrollable text area showing the device's internal log output. Useful for debugging: you can see incoming push payloads, API connection events, parse errors, and voice pipeline state transitions in real time.

This screen is not intended for regular use — it's a debugging tool. The text is rendered as a LVGL text area with a monospace-style font and auto-scrolls to the latest entry.

## Switches & actions panel

A button in the top-right area of the home screen (Home Assistant icon) swaps the bottom weather/forecast section for a **quick-action panel** of 5 cards:

| Card | Action |
|------|--------|
| PC Bureau | Calls `script.allumer_pc_tv` (PC + TV toggle) |
| Volet | Open / close / stop the roller shutter — smart: if the shutter is moving, the tap stops it; otherwise it opens or closes based on current target state |
| Chambre | Toggle bedroom light (`light.toggle`) |
| Salon | Toggle living room light (`light.toggle`) |
| LEDs Bureau | Calls `script.allumer_leds` (office LEDs) |

Tapping the same button again restores the forecast panel. The switch is stateful — the last active forecast slide (hourly or daily, and which page) is remembered and restored.

## Long press — detail popups

Some cards on the forecast slides support two interaction levels:

- **Short tap** — quick action (e.g., toggle the light directly)
- **Long press** — opens a **fullscreen modal** with detailed controls

Currently implemented for light entities on slide 3:

**Light popup (960×520 modal):**
- Left column: **brightness arc** (0–255) — drag to adjust, calls `script.tab5_set_light` in real time
- Right column: **9 color presets** in a 3×3 grid (Blanc, Chaud, On/Off / Rouge, Vert, Bleu / Rose, Orange, Cyan) — each sends a `light.turn_on` with the corresponding `color_name`
- The power button in the grid reflects the current light state (amber = on, dim = off)
- Tapping the dark overlay or the × button closes the modal

The popup is context-aware: the entity it controls is set dynamically at open time (`current_light_entity` global), so the same popup component handles multiple light entities without duplication.

## Color coding for readability

Color is used consistently throughout the interface as a primary information channel — not for decoration, but to let you read state at a glance without reading labels:

**Temperatures (all screens):** mapped to a continuous scale via `get_temperature_color()`
- Blue (cold) → green (comfortable) → orange (warm) → red (hot)
- Applied identically to indoor sensors, hourly forecast temperatures, and daily max/min

**Day names on daily forecast:**
- Cyan — today
- Green — rest day / day off
- Amber — Sunday (rest)
- Red — Sunday (worked)
- Rose — early shift (start before 09:00)
- Dim slate — past day (reduced opacity)

**Plant moisture bars:**
- Red — very dry (< 14%)
- Orange — dry
- Green — optimal (30–80%)
- Blue — too wet (> 80%)

**Microphone icon:**
- Grey (dim) — wake word off
- Grey — idle listening
- Green — recording
- Orange — processing
- Blue — speaking
- Red — error

**Rainfall in hourly forecast:** blue if precipitation > 0 mm, dim grey otherwise.

All color constants live in the `UIColor` namespace in `tab5_custom.h` and their YAML counterparts in `tab5-styles.yaml`. Changing a color in one place updates it everywhere it's used.

## Roller shutter control

The roller shutter is accessible both from the action panel (quick open/close/stop) and via the dedicated card on forecast slide 3. The volet card has smart toggle logic: a `volet_en_mouvement` global tracks whether the shutter is currently moving. A tap while it moves sends `stop`; a tap while stopped sends `open` or `close` based on the `volet_target_open` global (which HA keeps updated via push).

## Écran d'accueil

- Température et humidité intérieure (capteur salon)

## Weather screen (French)

L'écran météo est navigué par **swipe horizontal**, avec un **système de pagination à 5 slides** (indiqué par 5 points en bas de l'écran). Toutes les slides partagent la même mise en page : 5 cartes affichées côte à côte dans la partie basse de l'écran.

### Slides 1–2 : Prévisions horaires (15 tranches)

Les deux premières slides couvrent la **météo heure par heure** pour les 15 prochaines tranches, 5 par slide.

Chaque carte affiche :
- **Heure** (ex: `14:00`, `15:00`)
- **Icône de condition météo** — système d'icônes double couche : condition principale (nuage, soleil, lune, brouillard, vent) + superposition précipitation (pluie, fortes pluies, neige, grêle, orage) composited depuis la police `IconeMeteo.ttf`
- **Température** — avec code couleur : bleu (froid) → vert → orange → rouge (chaud)
- **Pluie** — précipitations en mm si > 0, `-` si sec

### Slides 3–5 : Prévisions journalières 15 jours

Les trois dernières slides couvrent les **prévisions sur 15 jours**, 5 jours par slide.

Chaque carte affiche :
- **Nom du jour** — code couleur selon le type :
  - Cyan = aujourd'hui
  - Vert = jour de repos
  - Ambre/Rouge = dimanche (travaillé ou pas)
  - Rose = prise de service matinale (départ avant 09:00)
  - Ardoise estompée = jour passé (opacité réduite)
- **Icône de condition météo** (même système double couche qu'horaire)
- **Températures max / min** — toutes deux avec code couleur individuel

Taper sur une carte jour **bascule** entre la vue météo et une vue détail des horaires d'ouverture (configuré par jour dans le payload push HA).

**Sur la slide 3 seulement (aujourd'hui + 4 prochains jours) :** chaque carte affiche aussi un bouton d'action raccourci (PC/TV, volet roulant, lumières) — ces boutons disparaissent sur les slides 4 et 5 pour garder les cartes lisibles.

**Source de données :** intégration Météo-France dans Home Assistant. HA sérialise le tableau complet 15 jours en un seul payload délimité par des points-virgules par mise à jour.

## Écran climatisation

Contrôle le climatiseur de la pièce via l'entité `climate` de Home Assistant.

**Ce qui est affiché :**
- Température ambiante actuelle (depuis un capteur de température dédié)
- Température cible — affichée comme grand nombre au centre d'un contrôle en arc
- Mode de fonctionnement du clim (chauffage / refroidissement / ventilation / auto / arrêt) — affiché comme label de mode avec icône correspondante

**Ce qu'on peut faire :**
- **Arc thermostat** — faire glisser l'indicateur d'arc pour régler la température cible. L'arc s'anime en douceur et envoie la nouvelle cible à Home Assistant en temps réel.
- **Boutons de mode** — taper pour basculer entre chauffage, refroidissement, auto et arrêt. Le bouton du mode actif est mis en surbrillance.

L'écran climatisation reste grisé (tous les contrôles estompés, non cachés) quand le clim est éteint, préservant la mise en page tout en communiquant clairement que l'unité est inactive.

## Écran plantes

Surveille jusqu'à cinq capteurs d'humidité du sol BLE (compatibles avec les capteurs standard type Xiaomi/MiFlora).

**Ce qui est affiché par emplacement de capteur :**
- Label du nom de la plante (personnalisable dans le bloc substitutions)
- Pourcentage d'humidité du sol — affiché comme barre de progression horizontale
- Couleur de la barre selon le niveau d'humidité :
  - 🔴 Rouge — très sec (< 20%) — besoin d'arrosage
  - 🟠 Orange — sec (20–35%)
  - 🟢 Vert — optimal (35–65%)
  - 🔵 Bleu — trop humide (> 65%)
- Température actuelle du capteur de zone plantes

Les capteurs BLE sont interrogés par le co-processeur ESP32-C6. Les données transitent via le proxy BLE ESPHome vers Home Assistant, puis sont renvoyées à l'écran via l'automatisation push standard. Pas de chemin BLE direct vers l'écran.

## Écran planning

Affiche les prochains événements de calendrier depuis Google Calendar.

**Ce qui est affiché :**
- Jusqu'à 4 emplacements d'événements
- Chaque emplacement montre :
  - Titre de l'événement (tronqué si trop long)
  - Heure de début ou date
  - Tag de couleur correspondant à la couleur de l'événement Google Calendar

Les tags de couleur utilisent la syntaxe de couleur inline de LVGL — le template Home Assistant lit la couleur de l'événement depuis l'API Google Calendar et l'embarque comme code hex directement dans la chaîne payload push. L'appareil la rend sans aucune logique de couleur supplémentaire.

Les données se mettent à jour sur un déclencheur `calendar.sync`, ou quand l'état de l'entité calendrier change. L'écran planning ne se rafraîchit pas sur un timer — il attend un push.

## Assistant vocal

L'icône microphone sur l'écran d'accueil est l'interface visuelle de l'assistant vocal. Elle change de couleur pour refléter l'état courant du pipeline :

| Couleur de l'icône | État | Ce qui se passe |
|-------------------|------|----------------|
| **Gris sombre** | En veille | Écoute wake-word désactivée |
| **Gris** | Repos | Écoute "Ok Nabu" en arrière-plan |
| **Vert** | Écoute | Wake word détecté — enregistrement en cours |
| **Orange** | Traitement | Pipeline HA exécute STT + reconnaissance d'intention |
| **Bleu** | Synthèse | Réponse TTS en lecture sur le haut-parleur |
| **Rouge** | Erreur | Pipeline échoué ou commande non comprise |

**Bascule wake-word :** un bouton sur l'écran d'accueil active ou désactive la détection du wake-word "Ok Nabu". Quand désactivé, on peut toujours activer l'assistant en tapant directement sur l'icône microphone (push-to-talk). L'icône passe en gris sombre quand la détection est coupée.

**Sélecteur de mode :** à côté de l'icône microphone, un petit bouton bascule entre deux modes :
- 🏠 **Mode Home Assistant** — les commandes vont vers l'agent de conversation standard de HA (contrôler les lumières, vérifier les capteurs, déclencher les automations)
- 🤖 **Mode Conversation** — les commandes vont vers un pipeline basé sur un LLM pour la conversation libre

Le mode est sauvegardé entre les redémarrages via l'entité HA `select` (`select.m5stack_tab5_home_assistant_hmi_assistant`).

## Écran console

Une zone de texte défilable montrant la sortie log interne de l'appareil. Utile pour le débogage : on peut voir les payloads push entrants, les événements de connexion API, les erreurs de parsing, et les transitions d'état du pipeline vocal en temps réel.

Cet écran n'est pas destiné à un usage régulier — c'est un outil de débogage. Le texte est rendu comme zone de texte LVGL avec une police style monospace et défile automatiquement vers la dernière entrée.

## Panel interrupteurs & actions

Un bouton en haut à droite de l'écran d'accueil (icône Home Assistant) **remplace le bas de l'écran météo/prévisions par un panel d'actions rapides** de 5 cartes :

| Carte | Action |
|-------|--------|
| PC Bureau | Appelle `script.allumer_pc_tv` (toggle PC + TV) |
| Volet | Ouvrir / fermer / stopper le volet — smart : si le volet est en mouvement, le tap l'arrête ; sinon il ouvre ou ferme selon l'état cible courant |
| Chambre | Bascule la lumière chambre (`light.toggle`) |
| Salon | Bascule la lumière salon (`light.toggle`) |
| LEDs Bureau | Appelle `script.allumer_leds` |

Retouch sur le même bouton restaure le panel prévisions. Le basculement est étaté — la dernière slide active (horaire ou journalière, et quelle page) est mémorisée et restaurée.

## Appui long — popups de détail

Certaines cartes des slides prévisions supportent deux niveaux d'interaction :

- **Appui court** — action rapide (ex: bascule la lumière directement)
- **Appui long** — ouvre un **popup modal plein écran** avec contrôles détaillés

Actuellement implémenté pour les entités lumière sur la slide 3 :

**Popup lumière (modal 960×520) :**
- Colonne gauche : **arc de luminosité** (0–255) — à glisser pour ajuster, appelle `script.tab5_set_light` en temps réel
- Colonne droite : **9 couleurs prédéfinies** en grille 3×3 (Blanc, Chaud, On/Off / Rouge, Vert, Bleu / Rose, Orange, Cyan) — chacune envoie un `light.turn_on` avec le `color_name` correspondant
- Le bouton On/Off dans la grille reflète l'état courant de la lumière (ambre = allumée, gris = éteinte)
- Taper l'overlay sombre ou le bouton × ferme le modal

Le popup est contextuel : l'entité qu'il contrôle est définie dynamiquement à l'ouverture (globale `current_light_entity`), donc le même composant popup gère plusieurs entités lumière sans duplication.

## Contrôle du volet roulant

Le volet est accessible à la fois depuis le panel d'actions (ouvrir/fermer/stopper en un tap) et via la carte dédiée sur la slide 3 des prévisions. La carte volet a une logique de toggle intelligente : une globale `volet_en_mouvement` suit si le volet est en mouvement. Un tap pendant le mouvement envoie `stop` ; un tap à l'arrêt envoie `open` ou `close` selon la globale `volet_target_open` (maintenue à jour par HA via push).
